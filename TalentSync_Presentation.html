02 — Problem Statement

Why Manual Screening Fails

The real cost of traditional resume filtering in modern recruitment

⏱️
Painfully Slow Process
Recruiters spend 6–8 seconds per resume. For 500 applications, that's hours of repetitive work with high cognitive load.
🔤
Keyword Matching is Broken
Searching "Java" misses "Spring Boot" experts. Semantic meaning is completely lost in traditional ATS systems.
🎯
Missing Great Candidates
Qualified candidates get filtered out due to phrasing differences — not skill gaps. A huge loss for both sides.
📚
Volume Overload
High-volume roles receive 200–1000+ applications. No human can fairly evaluate every resume without bias or fatigue.
🔒
No Visibility Control
No system to separate public talent pools from private company-specific resume databases with proper access control.
📉
Poor Ranking Logic
Traditional ATS ranks by keyword count, not actual relevance. Best candidates are often buried on page 3.

07 — Working Methodology

Step-by-Step System Flow

1
Recruiter Authentication
Signup/Login via bcrypt-hashed passwords. Server issues JWT access + refresh tokens stored in Streamlit session state.
auth_routes.py
2
Resume Upload (PDF/DOCX)
Multiple files uploaded via multipart form to POST /resumes/upload. Saved to data/raw_resumes/ folder.
resume_routes.py
3
Text Extraction
pdfplumber for PDFs, python-docx for DOCX. Raw text cleaned with regex to remove extra whitespace.
utils.py → extract_text()
4
Resume Parsing
Regex-based parser extracts name (first proper-name line), skills (keyword list of 50+ tech terms), experience (job title + year patterns).
llm_parser.py → parse_resume()
5
Embedding Generation
Resume text converted to a 384-dimensional vector using all-MiniLM-L6-v2. Normalized for cosine similarity. Runs 100% locally.
embedding.py → generate_embedding()
6
Dual Storage
Embedding stored in ChromaDB (persistent on disk). Structured data (name, skills, experience) stored in MongoDB Atlas.
chroma_client.py + database.py
7
JD Vector Search
JD text embedded → queried against ChromaDB using cosine similarity. Returns top 10 most similar resume IDs instantly.
jd_matcher.py → match_jd()
8
CrossEncoder Reranking
Each (JD, resume) pair scored by ms-marco-MiniLM-L-6-v2. Final results sorted by rerank score and returned with % match.
reranker.py → rerank()

13 — Evaluation

Advantages, Limitations & Future Scope

✅ Advantages
🧠Semantic understanding beyond keyword matching — finds hidden talent
💰Zero OpenAI cost — runs 100% locally with open-source Sentence Transformers
💾Persistent ChromaDB — embeddings survive backend restarts
🔐Secure JWT auth with bcrypt password hashing
📊Real-time dashboard with live MongoDB data, not hardcoded
🔌Decoupled frontend/backend — deployed independently on Streamlit Cloud + Render
🌐Fully live on cloud — accessible from anywhere, no local setup needed
⚠️ Limitations
🔤Parser uses regex — may miss skills in unusual resume formats
🔄Resumes must be re-uploaded if ChromaDB is manually cleared
📧No email notifications for recruiter activity
🔑OpenAI parsing (more accurate) requires paid API credits
📱No mobile-responsive design (Streamlit limitation)
☁️ChromaDB resets on Render restart — embeddings need re-uploading
🚀 Future Enhancements
🤖GPT-4o parsing for 100% accurate resume extraction when credits available
📧Email notifications for new matches and uploads
💬Explainable AI — reason why each candidate scored high
🌍Multilingual resume support for global hiring
🐳Docker containerization + persistent disk storage on cloud
📊Advanced analytics — hiring trends, skill gap analysis
